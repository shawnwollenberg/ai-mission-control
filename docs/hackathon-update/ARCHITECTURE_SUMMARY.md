# Architecture Summary

Mission Control is the control plane for an AI organization. A human supplies an objective; Mission Control decomposes and tracks the work, assigns bounded capabilities, exposes progress and evidence, and stops at decisions that require human authority. Individual agents are interchangeable organization members rather than the product’s source of truth.

The system is an event-sourced modular monolith. Its canonical append-only PostgreSQL event log records every meaningful business fact: mission launches, task assignments, execution changes, progress, approvals, artifacts, schedules, notifications, and outcomes. User-facing state is built from deterministic projections of that history. This means the mission plan, operational dashboard, audit views, and debrief can be rebuilt from an empty projection store. External side effects use an outbox and idempotency controls; replay restores recorded facts but never repeats a GitHub write, notification, or other external action.

A mission is broken into objectives and executable tasks. An execution records one bounded attempt to perform a task. Eligibility checks consider readiness, assignment, capabilities, permissions, policy, concurrency, emergency controls, and existing claims before work is exposed. Terminal transitions are monotonic, and message identifiers prevent duplicated network requests from duplicating business effects.

Agent-specific behavior lives behind adapters. Core events describe assignments, execution lifecycle, evidence, artifacts, and outcomes without embedding Codex-, Hermes-, or Claude-specific fields. Codex is the first production-complete local adapter. Hermes can participate in live coordination and execution through its existing integration, while additional local adapters can adopt the same transport and lifecycle contract without changing the control plane.

Mission Agent solves the localhost, NAT, and firewall problem with a pull model. It is a small local runtime launched by a versioned, checksummed command. The owner first creates a credential in Mission Control. Mission Agent stores that credential in the macOS Keychain when available or an owner-readable `0600` file on Linux, then initiates every connection over outbound HTTPS. It sends signed heartbeats, requests eligible assignments, acknowledges a claim, renews a lease, reports progress, checks cancellation, submits artifacts, and reports completion or failure. Leases allow safe recovery when a process exits, and persisted local state supports restart recovery. Mission Control never opens a tunnel or sends arbitrary shell commands to the user’s machine.

The first mission uses a deliberately constrained Codex operation: read-only repository analysis. The adapter receives an explicit repository selected by the owner, runs Codex with reduced environment access, and verifies repository state before and after execution. If files change, the execution stops instead of pretending the read-only contract held. The resulting Markdown report is uploaded as an artifact and linked to its execution and canonical events.

Policies and approvals are separate from agent intent. Deterministic policy evaluation can deny an action, allow a bounded operation, or create a contextual request for a human. An approval records the exact proposed effect and evidence. Agents have no autonomous authority to merge, deploy, modify infrastructure or secrets, remediate production systems, sign or submit transactions, or move assets. GitHub branch push and pull-request creation are bounded, auditable effects; merge remains unavailable.

Background workers project events, dispatch outbox effects, evaluate schedules, deliver notifications, recover expired work, and surface dead letters. Worker health and usage/budget projections feed the operations dashboard. Artifacts preserve the evidence produced by an execution, while presentation sanitization prevents local paths or private repository details from leaking into public views.

Workspace isolation is enforced at authentication, command validation, assignment selection, event projection, and artifact access. Every new account receives a private personal workspace and owner role. Credentials are scoped to the owner-created agent and workspace, may be rotated or revoked, and cannot discover work from another tenant.

Production runs on a single low-cost AWS Graviton instance with the web application, workers, reverse proxy, and PostgreSQL containerized together, plus object storage for artifacts. This intentionally modest topology avoids a managed database, load balancer, or redundant instances while adoption is early. It can later be separated along existing boundaries without changing the event model or pull protocol.
